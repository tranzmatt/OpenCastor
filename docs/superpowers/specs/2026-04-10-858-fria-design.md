# Design: FRIA CLI — `castor fria generate` (craigm26/OpenCastor#858)

**Date:** 2026-04-10
**Issue:** craigm26/OpenCastor#858
**Related spec:** continuonai/rcan-spec#195 (§22)
**Status:** Approved — pending implementation plan
**Scope:** craigm26/OpenCastor · continuonai/rcan-spec

---

## 1. Problem Statement

EU AI Act Art. 27 requires deployers of high-risk AI systems to conduct a Fundamental Rights Impact Assessment (FRIA) before deployment. Annex IV specifies the technical file that must accompany a conformity assessment submission to a notified body. OpenCastor has a conformance checker, ML-DSA-65 robot identity, and a robot memory system — but no way to assemble these into a signed, citable artifact a notified body can verify.

This design closes that gap: a single CLI command (`castor fria generate`) produces a signed JSON artifact and a print-ready HTML companion document, both citable in a conformity assessment technical file. The document format is defined in a new rcan-spec §22 so the artifact can cite a stable public specification URL (`rcan.dev/spec/section-22`).

---

## 2. Approach

**Dedicated `castor/fria.py` module** — mirrors the `castor/watermark.py` pattern from #857. One module owns all FRIA logic: prerequisite check, document assembly, signing, and HTML rendering. The CLI handler (`cmd_fria_generate` in `castor/cli.py`) calls the module. Clean boundary, independently testable.

---

## 3. Architecture

Four components across two repos:

| Component | Repo | Responsibility |
|---|---|---|
| `castor/fria.py` | OpenCastor | `build_fria_document()`, `sign_fria()`, `render_fria_html()` |
| `castor/templates/fria.html.j2` | OpenCastor | Jinja2 print-ready HTML template |
| `castor/cli.py` | OpenCastor | `castor fria generate` subcommand → `cmd_fria_generate()` |
| `src/pages/spec/section-22.astro` | rcan-spec | §22 FRIA Protocol definition |

Data flows in one direction: CLI → `fria.py` (build → prerequisite check → sign → render) → writes `fria.json` + `fria.html`.

## 4. CLI Interface

```bash
castor fria generate \
  [--config bot.rcan.yaml]       # RCAN config file; default: auto-detect
  [--output fria.json]           # JSON output path; default: fria-{rrn}-{date}.json
  [--html fria.html]             # HTML output path; default: same stem as --output
  [--annex-iii <basis>]         # Annex III classification basis
  [--intended-use "..."]         # Free-text deployment description
  [--force]                      # Bypass conformance prerequisite gate
  [--no-html]                    # JSON artifact only, no HTML companion
  [--skip-sign]                  # Omit ML-DSA-65 signature (dev/test mode)
```

**Valid `--annex-iii` values:**

```
safety_component          # Safety components of products (Annex III §1)
biometric                 # Biometric identification (Annex III §2)
critical_infrastructure   # Critical infrastructure (Annex III §3)
education                 # Education and vocational training (Annex III §4)
employment                # Employment and workers management (Annex III §5)
essential_services        # Access to essential services (Annex III §6)
law_enforcement           # Law enforcement (Annex III §7)
migration                 # Migration, asylum, border control (Annex III §8)
administration_of_justice # Administration of justice (Annex III §9)
general_purpose_ai        # General-purpose AI (Annex III §10)
```

**Conformance prerequisite gate:**

Before generating, `cmd_fria_generate` runs `ConformanceChecker(config).run_all()` and verifies:
- Conformance score ≥ 80/100
- Zero `safety.*` failures

If either condition fails, the command exits non-zero and prints the blocking check IDs with their `fix` strings. `--force` bypasses the gate and adds `"prerequisite_waived": true` to the document (visible to the notified body).

## 5. FRIA JSON Document Schema

```json
{
  "schema": "rcan-fria-v1",
  "spec_ref": "https://rcan.dev/spec/section-22",
  "generated_at": "2026-04-10T14:32:01.123Z",
  "system": {
    "rrn": "RRN-000000000001",
    "rrn_uri": "rrn://craigm26/robot/opencastor-rpi5-hailo/bob-001",
    "robot_name": "bob",
    "opencastor_version": "2026.4.10.0",
    "rcan_version": "1.9.0",
    "agent_provider": "google",
    "agent_model": "gemini-1.5-flash"
  },
  "deployment": {
    "annex_iii_basis": "safety_component",
    "intended_use": "indoor autonomous navigation",
    "prerequisite_waived": false
  },
  "conformance": {
    "score": 87,
    "pass": 24,
    "warn": 2,
    "fail": 0,
    "checks": [
      {
        "check_id": "safety.estop_configured",
        "category": "safety",
        "status": "pass",
        "detail": "Emergency stop configured"
      },
      {
        "check_id": "safety.confidence_gates_configured",
        "category": "safety",
        "status": "warn",
        "detail": "Confidence gate present but threshold below recommended 0.7",
        "fix": "Set safety.confidence_threshold >= 0.7 in config"
      }
    ]
  },
  "human_oversight": {
    "hitl_configured": true,
    "confidence_gates_configured": true,
    "estop_configured": true
  },
  "hardware_observations": [
    {
      "id": "mem-abc123",
      "text": "Left motor stalls at 50% duty cycle under load",
      "confidence": 0.82,
      "tags": ["motor", "failure-mode"]
    }
  ],
  "signing_key": {
    "alg": "ml-dsa-65",
    "kid": "<key-id>",
    "public_key": "<base64url-encoded ML-DSA-65 public key>"
  },
  "sig": {
    "alg": "ml-dsa-65",
    "kid": "<key-id>",
    "value": "<base64url signature over canonical JSON with sig field absent>"
  }
}
```

**Field sources:**

| Field | Source |
|---|---|
| `system.*` | `config["metadata"]`, `rcan.__version__`, `config["agent"]` |
| `deployment.annex_iii_basis` | `--annex-iii` CLI argument |
| `deployment.intended_use` | `--intended-use` CLI argument |
| `conformance.*` | `ConformanceChecker(config).run_all()` + `.summary()` |
| `human_oversight.*` | Derived from conformance check results for `safety.hitl_*`, `safety.confidence_gates_*`, `safety.estop_*` |
| `hardware_observations` | `load_memory(memory_path)` filtered to `HARDWARE_OBSERVATION` with confidence ≥ 0.30 |
| `signing_key` | `get_message_signer(config).public_key_bytes()` + `signer._pq_key_id` |
| `sig` | ML-DSA-65 signature over `json.dumps(doc_without_sig, sort_keys=True, separators=(',', ':'))` |

**Signing:** The signature covers canonical JSON (no whitespace, keys sorted, `sig` field absent). Same pattern as `sign_message_dict()` in `castor/rcan/message_signing.py`. When `--skip-sign` is passed, the `sig` field is omitted entirely.

## 7. HTML Template (`castor/templates/fria.html.j2`)

Print-ready layout with inline CSS only (no external resources — renders offline, prints cleanly). Sections:

1. **Cover page** — document title "Fundamental Rights Impact Assessment", RRN, robot name, Annex III basis, generated date, spec citation (`rcan.dev/spec/section-22`)
2. **System identity table** — all `system.*` fields
3. **Deployment classification** — Annex III basis, intended use, prerequisite_waived warning banner if true
4. **Conformance evidence** — score gauge, pass/warn/fail counts, full check table color-coded (green/amber/red)
5. **Human oversight** — HiTL, confidence gates, ESTOP as pass/fail indicators
6. **Hardware observations** — table of HARDWARE_OBSERVATION entries with confidence bars
7. **Signing key fingerprint** — algorithm, key ID, truncated public key
8. **Footer** — `Generated by OpenCastor {version} · Specification: rcan.dev/spec/section-22 · EU AI Act Art. 9, Art. 27, Annex IV`

## 8. rcan-spec §22

New `src/pages/spec/section-22.astro` page. Status badge: `v1.0 Draft`.

**Page structure:**
- H2: "§22 — Fundamental Rights Impact Assessment (FRIA) Protocol"
- H3: "Overview" — what a FRIA is, EU AI Act Art. 27 context, when required (L2+ conformance deployments)
- H3: "Document Schema" — full JSON field reference table (field, type, required, source, description)
- H3: "Conformance Prerequisite" — score ≥ 80, 0 safety failures; `prerequisite_waived` semantics
- H3: "Signing" — ML-DSA-65 canonical JSON signing (same algorithm as §16.5 watermarking); signature verification
- H3: "HTML Companion" — format description, rendering requirements
- H3: "EU AI Act Mapping" — table: FRIA document section → EU AI Act article
- H3: "CLI Reference" — `castor fria generate` usage
- Footer: links to §16.5 (watermarking), rcan.dev/compliance/frameworks, OpenCastor #858

**EU AI Act Mapping table:**

| FRIA Section | EU AI Act Provision |
|---|---|
| `conformance` block | Art. 9 — Risk management system |
| `system` block | Art. 13 — Transparency and information |
| `human_oversight` block | Art. 14 — Human oversight |
| `hardware_observations` | Art. 17 — Quality management system |
| Full document + `sig` | Art. 27 — FRIA obligation |
| `spec_ref` citation | Annex IV — Technical documentation |

## 10. Out of Scope

- PDF generation (HTML + browser print-to-PDF is sufficient for notified body submission)
- rcan-py or rcan-ts SDK surface for FRIA (follow-on if needed)
- FRIA verification endpoint (the signed JSON is self-verifying via `signing_key.public_key`)
- Automated submission to notified body registries (manual step)
- rcan-spec §22 as a full dedicated page beyond the subsections above
